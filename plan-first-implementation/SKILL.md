# 플랜 퍼스트 구현

이 스킬은 `문서 먼저, 구현 나중`을 강하게 적용할 때 쓴다. 계획 문서가 이번 단계의 source of truth이고, 구현 가능한 작업에서는 계획 문서가 실제 실행 산출물로 이어지는 상위 인덱스 역할을 한다.

전역 스킬셋 안에서 이 스킬은 `주력 계획 스킬`로 취급한다. 유지 기준과 다른 계획 스킬과의 관계는 [/Users/moonsoo/.codex/skills/README.md](/Users/moonsoo/.codex/skills/README.md)를 참고한다.

## 언제 쓸까

- 사용자가 plan first를 원한다.
- 큰 수정이라 근거와 범위를 먼저 잠가야 한다.
- 구현 전에 파일 관계와 리스크를 시각화해야 한다.
- 계획 MD만 만들면 실행 가능한 결과물이 남지 않는 작업이다.
- 화면, 흐름, 상태, 인터랙션, 작은 기능을 HTML로 먼저 구체화해야 하는 작업이다.

## Artifact decision gate

계획 문서를 쓰기 전에 이번 요청의 실행 산출물을 먼저 분류한다. 이 판정은 계획 문서의 `## 구현 산출물` 섹션과 최종 답변에 그대로 반영한다.

| 요청 유형 | 기본 산출물 | HTML 생략 가능 여부 |
| --- | --- | --- |
| `계획만`, `문서만`, 조사/리뷰/백엔드/보안/운영 계획 | 계획 MD | 가능. 단, `구현 산출물` 섹션에 생략 사유를 남긴다. |
| 화면, 흐름, 상태, 인터랙션, 폼, 대시보드, 목업, UX 비교 | 계획 MD + HTML artifact | 원칙적으로 불가. |
| 기존 앱 화면 직접 수정 | 계획 MD + dev server URL | HTML demo는 생략 가능하나 대체 URL과 생략 사유는 필수다. |
| 기존 앱 수정 전 독립 검토가 필요한 UI | 계획 MD + HTML artifact + 후속 앱 구현 계획 | 불가. |
| CLI, API, DB, 서버 로직 중심 | 계획 MD + 검증 명령/로그 | 가능. |

화면, 흐름, 상태, 인터랙션이 포함된 요청은 사용자가 명시적으로 `계획만`이라고 하지 않는 한 `계획 후 구현`으로 본다. 이때 단일 HTML artifact 또는 실행 중인 앱 URL 중 하나는 반드시 최종 답변에 있어야 한다.

## 산출물 계약

기본 산출물은 한 장짜리 계획 파일이 아니라 아래 계층이다.

1. 상위 계획 MD
   - 이번 작업의 source of truth다.
   - 범위, 근거, 리스크, 검증, 구현 순서, 관련 파일을 담는다.
   - 구현 산출물 링크를 반드시 가진다.
2. HTML 구현 산출물
   - 계획이 다루는 실제 기능, 화면, 흐름, 상태 변화를 작게라도 실행 가능하게 만든다.
   - 새 프로젝트나 독립 프로토타입이면 `docs/plans/artifacts/<plan-slug>.html` 또는 계획 문서 옆의 `<plan-slug>.html`에 둔다.
   - 기존 앱 코드가 주 구현 대상이면 실제 앱 파일을 고치고 dev server URL을 산출물로 둔다. 다만 사용자가 비교/검토/목업/시안/흐름 확정을 원했거나, 앱 구현 전에 상태 전환을 독립적으로 확인해야 하면 HTML demo도 필수다.
3. 계획 MD의 참조 링크
   - 계획 문서 안에 `## 구현 산출물` 섹션을 만들고 HTML 파일을 상대 링크로 건다.
   - 예: `- [HTML 기능 초안](artifacts/2026-05-10-feature-slice.html)`
4. 대화창 산출물 링크
   - 구현이 끝난 뒤 최종 답변에도 사용자가 바로 누를 수 있는 링크를 보낸다.
   - 계획 MD 안의 링크만으로 끝내지 않는다.
   - 단일 HTML이면 절대 파일 경로 링크를 쓴다. 예: `[HTML 기능 초안](/Users/moonsoo/project/docs/plans/artifacts/demo.html)`
   - dev server가 필요한 앱이면 서버를 실행한 뒤 URL을 쓴다. 예: `[로컬 미리보기](http://localhost:5173/path)`

사용자가 명시적으로 `계획만` 원하면 HTML을 만들지 않는다. 그 대신 계획 문서의 `구현 산출물` 섹션에 어떤 HTML을 만들지와 왜 지금 생략하는지를 적는다.

## 핵심 흐름

1. Artifact decision gate로 이번 요청이 `계획만`, `계획 후 구현`, `HTML 필수`, `앱 URL 대체`, `HTML 생략 가능` 중 어디에 해당하는지 먼저 정한다.
2. 기존 계획 문서가 있으면 이어서 쓰고, 없으면 새 상위 계획 MD를 만든다.
3. 관련 코드와 문서를 읽어 현재 동작, 변경 지점, 검증 표면을 정리한다.
4. 디자인, UX, 카피, 정보 구조처럼 방향 선택이 필요한 작업이면 계획 문서를 쓰기 전에 짧은 질문으로 방향을 잠근다. 질문은 한 번에 하나씩 하고, 사용자가 바로 고를 수 있는 선택지나 대비를 주는 편을 기본으로 한다.
5. 계획 문서에 범위, 근거, 리스크, 검증, 구현 순서, 관련 파일, `구현 산출물` 섹션을 적는다.
6. 문서가 구현 직전 수준으로 준비됐을 때만 HTML 또는 실제 앱 코드 수정으로 넘어간다.
7. `계획 후 구현`이면 HTML 구현 산출물을 만든다.
8. 기존 앱 직접 수정이 산출물이라면 실행 중인 dev server URL을 계획 문서와 최종 답변에 연결한다.
9. HTML을 만들지 않았다면 `HTML 생략 보고서`를 계획 문서와 최종 답변에 남긴다.
10. HTML 또는 앱 URL을 브라우저에서 열 수 있는지, 중요한 인터랙션이 실제로 동작하는지 검증한다.
11. 최종 답변에 `산출물 링크:`를 넣고, 사용자가 대화창에서 바로 열 수 있는 HTML 파일 링크나 실행 중인 dev server URL을 제공한다.

## 근거 수집용 CLI 우선 규칙

- 파일 맵을 만들 때는 `fd`를 먼저 보고, 텍스트 근거를 모을 때는 `rg`를 먼저 쓴다.
- 구조화된 설정이나 워크플로 파일은 손으로 훑기보다 `yq`와 `jq`를 먼저 고려한다.
- 검증 명령은 임의 조합보다 repo의 `justfile` 유무를 먼저 확인한다.
- `bat`은 사람 눈으로 읽기 좋은 spot inspection에만 쓰고, 구조화 추출의 기본 도구로 쓰지 않는다.
- 자세한 매핑은 [CLI Core Tools Reference](/Users/moonsoo/projects/codex-skills-user/docs/cli-core-tools-reference-2026-03-24.md)를 따른다.

## HTML 구현 기준

- HTML은 장식용 목업이 아니라 최소 기능 단위여야 한다.
- 버튼, 입력, 토글, 탭, 필터, 상태 변경처럼 계획의 핵심 행동은 실제 JavaScript로 동작하게 한다.
- 외부 빌드가 필요 없는 단일 HTML을 우선한다. 기존 앱 맥락이 더 중요하면 repo의 프레임워크를 따르고 dev server URL을 산출물로 둔다.
- 데이터가 아직 없으면 작은 샘플 데이터를 HTML 안에 넣고 `mock`이라고 명시한다.
- 사용자가 inspect할 수 있게 레이아웃, 상태, 에러/빈 상태, 성공 상태 중 핵심 상태를 포함한다.
- 스타일은 과하게 꾸미기보다 기능 판단에 필요한 구조와 가독성을 우선한다.

## 계획 MD 템플릿 추가 항목

계획 문서에는 최소한 아래 섹션을 포함한다.

```md
## 구현 산출물

- HTML: [기능 초안](artifacts/<plan-slug>.html)
- 상태: planned | implemented | verified
- 실제 동작:
  - <HTML에서 실제로 동작하는 것>
- Mock:
  - <아직 샘플 데이터나 가짜 처리인 것>
```

HTML을 만들지 않는 경우에는 같은 섹션에 아래 블록을 반드시 넣는다.

```md
## HTML 생략 보고서

- 판정: 생략 가능 | 생략 불가
- 생략 사유:
  - <계획만/백엔드/기존 앱 URL 대체 등>
- 대체 산출물:
  - <앱 URL, 검증 명령, 테스트 로그, 문서 링크>
- 사용자가 바로 열어볼 링크:
  - <없으면 이유>
```

## 최종 답변 링크 규칙

계획 후 구현까지 끝낸 경우, 최종 답변에는 반드시 아래 중 하나를 넣는다.

```md
산출물 링크: [HTML 기능 초안](/absolute/path/to/artifacts/<plan-slug>.html)
```

또는 dev server가 필요한 경우:

```md
산출물 링크: [로컬 미리보기](http://localhost:<port>/<route>)
```

규칙:

- 링크는 사용자가 대화창에서 바로 누를 수 있어야 한다.
- 정적 HTML은 `file://` 대신 절대 경로 Markdown 링크를 쓴다.
- dev server가 필요한 앱은 가능하면 서버를 실제로 켠 뒤 URL을 준다.
- 서버를 켤 수 없으면 실행 명령, 예상 URL, 막힌 이유를 함께 말한다.
- 계획 문서 링크와 산출물 링크를 둘 다 주되, 산출물 링크를 먼저 둔다.
- HTML을 만들지 않았다면 최종 답변에 `HTML 생략 사유:`를 반드시 쓴다. 이 줄이 없으면 산출물 계약을 완료한 것으로 보지 않는다.

최종 답변에는 아래 네 줄을 우선 포함한다.

```md
- 계획 MD:
- 산출물 링크:
- HTML 생략 사유: <생략하지 않았으면 `해당 없음`>
- 검증 상태:
```

## 가드레일

- 계획이 덜 끝났는데 습관적으로 구현부터 하지 않는다.
- 근거 없는 추측을 계획 문서에 사실처럼 적지 않는다.
- 사용자가 계획만 원하면 구현으로 넘어가지 않는다.
- `계획 후 구현` 요청에서 계획 MD만 만들고 멈추지 않는다.
- HTML 구현 산출물을 만들었다면 상위 계획 MD 링크를 빠뜨리지 않는다.
- HTML 구현 산출물을 만들었다면 최종 답변의 대화창 링크를 빠뜨리지 않는다.
- HTML 구현 산출물을 만들지 않았다면 계획 문서와 최종 답변의 `HTML 생략 사유`를 빠뜨리지 않는다.
- 링크만 걸고 HTML 안의 핵심 인터랙션이 비어 있는 상태를 완료로 보지 않는다.
- 여러 질문을 한꺼번에 던져 사용자를 피곤하게 만들지 않는다. 방향을 가르는 질문부터 한 개씩 잠근다.
- 사용자가 좋아한 질문 방식이나 결정 패턴이 드러나면, 같은 세션에서는 그 상호작용 스타일을 유지한다.
