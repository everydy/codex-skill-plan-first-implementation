# 플랜 퍼스트 구현

이 스킬은 `문서 먼저, 구현 나중`을 강하게 적용할 때 쓴다. 계획 문서가 이번 단계의 source of truth라고 보고 움직인다.

전역 스킬셋 안에서 이 스킬은 `주력 계획 스킬`로 취급한다. 다른 계획 스킬이 함께 있다면, 이 스킬은 코드 수정 전 계획 문서를 먼저 잠그는 기본 흐름을 담당한다.

## 언제 쓸까

- 사용자가 plan first를 원한다.
- 큰 수정이라 근거와 범위를 먼저 잠가야 한다.
- 구현 전에 파일 관계와 리스크를 시각화해야 한다.

## 핵심 흐름

1. 이번 요청이 `계획만`인지 `계획 후 구현`인지 먼저 정한다.
2. 기존 계획 문서가 있으면 이어서 쓰고, 없으면 새 문서를 만든다.
3. 관련 코드와 문서를 읽어 현재 동작, 변경 지점, 검증 표면을 정리한다.
4. 디자인, UX, 카피, 정보 구조처럼 방향 선택이 필요한 작업이면 계획 문서를 쓰기 전에 짧은 질문으로 방향을 잠근다. 질문은 한 번에 하나씩 하고, 사용자가 바로 고를 수 있는 선택지나 대비를 주는 편을 기본으로 한다.
5. 계획 문서에 범위, 근거, 리스크, 검증, 구현 순서, 관련 파일을 적는다.
6. 문서가 구현 직전 수준으로 준비됐을 때만 코드 수정으로 넘어간다.

## 근거 수집용 CLI 우선 규칙

- 파일 맵을 만들 때는 `fd`를 먼저 보고, 텍스트 근거를 모을 때는 `rg`를 먼저 쓴다.
- 구조화된 설정이나 워크플로 파일은 손으로 훑기보다 `yq`와 `jq`를 먼저 고려한다.
- 검증 명령은 임의 조합보다 repo의 `justfile` 유무를 먼저 확인한다.
- `bat`은 사람 눈으로 읽기 좋은 spot inspection에만 쓰고, 구조화 추출의 기본 도구로 쓰지 않는다.
- 로컬에 별도 CLI 도구 기준 문서가 있다면 그 기준을 따르되, 공개 기본값은 `fd`, `rg`, `jq`, `yq`, `just` 우선이다.

## 꼭 남겨야 할 것

- in-scope 시스템 그림
- 바뀌는 파일과 유지해야 할 동작
- 남은 미확정 사항
- 검증 방법

## 가드레일

- 계획이 덜 끝났는데 습관적으로 구현부터 하지 않는다.
- 근거 없는 추측을 계획 문서에 사실처럼 적지 않는다.
- 사용자가 계획만 원하면 구현으로 넘어가지 않는다.
- 여러 질문을 한꺼번에 던져 사용자를 피곤하게 만들지 않는다. 방향을 가르는 질문부터 한 개씩 잠근다.
- 사용자가 좋아한 질문 방식이나 결정 패턴이 드러나면, 같은 세션에서는 그 상호작용 스타일을 유지한다.
